Setting Up Network Sharing (Wi-Fi Hotspot) from Fedora 42 (Ethernet-connected)

This guide leverages NetworkManager, the default and recommended network management service on Fedora, to share an Ethernet-based internet connection over a Wi-Fi hotspot.
Core Concepts

    NetworkManager: Manages network connections, including creating Wi-Fi hotspots and handling connection sharing (NAT/Masquerade).

    IP Forwarding: A kernel setting that allows the Fedora system to route packets between different network interfaces (Ethernet and Wi-Fi hotspot).

    NAT (Network Address Translation) / Masquerade: A technique used by the kernel's firewall (iptables/nftables) to allow multiple devices on a private network (your hotspot clients) to share a single public IP address (from your Ethernet connection) to access the internet. NetworkManager automatically configures this when "shared" mode is selected.

    Firewalld: Fedora's dynamic firewall manager. While NetworkManager handles the NAT rules, Firewalld still needs to allow the necessary traffic.

Step-by-Step Guide
1. Verify and Prepare Your System

    Ensure Internet Connectivity via Ethernet:
    Before anything else, confirm your Fedora machine has internet access through its Cat5 Ethernet connection.
    code Bash

IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
ip a # Look for your ethX or enpXsY interface with an IP address
ping -c 4 google.com # Test connectivity

  

If you don't have internet, troubleshoot your Ethernet connection first.

Identify Your Wi-Fi Device Name:
You'll need the exact name of your wireless interface.
code Bash
IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
nmcli device status

  

Look for a device with wifi type (e.g., wlp0s20f3, wlan0). Let's use WIFI_IFACE as a placeholder for it.

Install Essential Packages (if not present):
NetworkManager and its applet are usually installed by default on Fedora Workstation. dnsmasq isn't strictly necessary as NetworkManager's shared mode uses its own internal DHCP/DNS, but it can be useful for advanced scenarios.
code Bash

    IGNORE_WHEN_COPYING_START
    IGNORE_WHEN_COPYING_END

        
    sudo dnf install NetworkManager-wifi # Ensures Wi-Fi support
    sudo dnf install network-manager-applet # For graphical configuration (Gnome/KDE)
    # sudo dnf install dnsmasq # Optional, not usually needed for basic hotspot sharing

      

2. Create the Wi-Fi Hotspot

This is the central step where you configure the Wi-Fi interface to act as an access point and share the internet.

A. Graphical Method (Recommended for ease of use)

This is the simplest way for most desktop environments using NetworkManager.

    Open Settings (Gnome) or System Settings (KDE).

    Navigate to Network (Gnome) or Connections/Network (KDE).

    Ensure Wi-Fi is turned On.

    Click the + button (or similar "Add" button) next to "Wi-Fi" or "Wireless".

    Select "Create Wi-Fi Hotspot...".

    A new window will appear. It usually pre-fills a suggested Network Name (SSID) and generates a password.

        Network Name (SSID): Choose a name for your hotspot (e.g., MyFedoraHotspot).

        Password: Set a strong WPA2 password.

    Click "Create".

    NetworkManager will configure the hotspot and enable sharing. You should see your new hotspot listed as "Hotspot" or by its SSID.

B. Command Line Method (using nmcli)

This provides more control and is useful for headless systems or scripting.

    Define Variables (optional but good practice):
    code Bash

IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
WIFI_IFACE="wlp0s20f3"       # Replace with your actual Wi-Fi interface name (e.g., wlan0, wlpXsY)
HOTSPOT_SSID="MyFedoraHotspot"
HOTSPOT_PASSWORD="YourVerySecurePassword"
HOTSPOT_CONN_NAME="Fedora Hotspot" # A name for NetworkManager to store this connection profile

  

Create the Wi-Fi Hotspot Connection Profile:
code Bash
IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
sudo nmcli connection add type wifi ifname ${WIFI_IFACE} con-name "${HOTSPOT_CONN_NAME}" autoconnect yes ssid "${HOTSPOT_SSID}"
sudo nmcli connection modify "${HOTSPOT_CONN_NAME}" wifi-sec.key-mgmt wpa-psk wifi-sec.psk "${HOTSPOT_PASSWORD}"
sudo nmcli connection modify "${HOTSPOT_CONN_NAME}" ipv4.method shared
sudo nmcli connection modify "${HOTSPOT_CONN_NAME}" ipv6.method shared # Optional, but good for dual-stack networks

  

    ipv4.method shared (and ipv6.method shared) is critical. This tells NetworkManager to:

        Enable IP forwarding for this connection.

        Run a DHCP server on the hotspot interface.

        Perform NAT/Masquerade for traffic going out the primary internet connection.

Activate the Hotspot:
code Bash

    IGNORE_WHEN_COPYING_START
    IGNORE_WHEN_COPYING_END

        
    sudo nmcli connection up "${HOTSPOT_CONN_NAME}"

      

3. Enable IP Forwarding

While ipv4.method shared usually handles this, it's good practice to ensure the kernel setting for IP forwarding is permanently enabled. This allows your system to route packets between interfaces.

    Permanently Enable IP Forwarding:
    Edit the sysctl.conf file:
    code Bash

IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
sudo nano /etc/sysctl.conf

  

Add or uncomment the following line:
code Code
IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
net.ipv4.ip_forward = 1
# If you also want IPv6 forwarding, add:
# net.ipv6.conf.all.forwarding = 1

  

Save the file (Ctrl+O, Enter, Ctrl+X if using nano).

Apply Changes Without Rebooting:
code Bash

    IGNORE_WHEN_COPYING_START
    IGNORE_WHEN_COPYING_END

        
    sudo sysctl -p

      

    You can verify it's active with sysctl net.ipv4.ip_forward. It should output net.ipv4.ip_forward = 1.

4. Firewall Configuration (Firewalld)

NetworkManager, when configured for shared mode, is designed to integrate with Firewalld. It typically adds the necessary MASQUERADE rules automatically to the zone of the internet-facing interface (your Ethernet enpXsY).

However, it's good to understand and verify.

    Identify Zones:
    code Bash

IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
sudo firewall-cmd --get-active-zones

  

    Your Ethernet interface (enpXsY) should ideally be in a zone like public or external.

    Your Wi-Fi hotspot interface (WIFI_IFACE) is often placed into a internal or home zone by NetworkManager when sharing is active, or a new nm-shared zone might be created.

Verify Masquerading (NAT) on the Internet-Facing Zone:
NetworkManager should handle this automatically. You can check if masquerading is enabled on your internet-facing zone (e.g., public):
code Bash
IGNORE_WHEN_COPYING_START
IGNORE_WHEN_COPYING_END

    
sudo firewall-cmd --zone=public --query-masquerade

  

If it returns yes, you're good. If no, you can add it (though NetworkManager failing to add it usually indicates a deeper issue):
code Bash

    IGNORE_WHEN_COPYING_START
    IGNORE_WHEN_COPYING_END

        
    sudo firewall-cmd --zone=public --add-masquerade --permanent
    sudo firewall-cmd --reload

      

    Note: If your Ethernet interface is in a different zone (e.g., home), replace public with that zone name.

5. Connect Client Devices

On your phone, laptop, or other devices:

    Scan for Wi-Fi networks.

    Select your hotspot's SSID (e.g., MyFedoraHotspot).

    Enter the WPA2 password you set.

    Your devices should now connect and receive an IP address from your Fedora system (e.g., 10.42.0.X or 10.42.0.1 is often the hotspot gateway), and be able to access the internet.

Common Troubleshooting Pain Points and Resolutions

Here are common issues encountered when setting up an internet-sharing hotspot and how to address them comprehensively:

    Pain Point: Clients Connect to Hotspot but No Internet Access

        Symptom: Devices connect to your hotspot (get an IP address) but cannot browse the web or ping 8.8.8.8.

        Root Causes:

            IP forwarding is not enabled.

            NAT (masquerading) rules are missing or incorrect in the firewall.

            Fedora system itself doesn't have internet access via Ethernet.

            DNS resolution issues on the client.

        Resolution:

            Verify Fedora Internet Access: ping -c 4 google.com on your Fedora machine. If this fails, fix your Ethernet connection first.

            Check IP Forwarding: Run sysctl net.ipv4.ip_forward. It must output 1. If not, re-run sudo sysctl -w net.ipv4.ip_forward=1 and ensure net.ipv4.ip_forward = 1 is in /etc/sysctl.conf and sudo sysctl -p has been run.

            Check Masquerading:

                Identify your Ethernet interface's zone: sudo firewall-cmd --get-active-zones. Let's say it's public.

                Check if masquerade is active for that zone: sudo firewall-cmd --zone=public --query-masquerade.

                If no, enable it: sudo firewall-cmd --zone=public --add-masquerade --permanent followed by sudo firewall-cmd --reload.

                Also, ensure that your hotspot interface is in a zone that allows traffic out to the public zone (e.g., internal or nm-shared which typically has forwarding enabled to other zones by NetworkManager).

            NetworkManager State: Restart NetworkManager: sudo systemctl restart NetworkManager. Sometimes a fresh start clears up issues.

            Client DNS: The "shared" mode uses NetworkManager's internal DNS proxy. Clients should automatically use your Fedora machine as their DNS server. If you suspect DNS issues, try ping 8.8.8.8 from the client (to bypass DNS) vs. ping google.com. If 8.8.8.8 works but google.com doesn't, it's a DNS issue. This is less common with NetworkManager's shared mode, but can be a symptom.

    Pain Point: Wi-Fi Hotspot Not Visible to Clients

        Symptom: Your clients cannot see the SSID broadcast by your Fedora machine.

        Root Causes:

            Wi-Fi adapter is disabled (hardware switch, rfkill).

            Wi-Fi driver issues.

            Hotspot connection profile is not active or failed to activate.

            Incorrect Wi-Fi channel or mode.

        Resolution:

            Check Hardware Switch/RFKill: Ensure any physical Wi-Fi switch on your laptop is ON. Check rfkill list in terminal. If anything is hard blocked or soft blocked, try sudo rfkill unblock all.

            Verify Wi-Fi Device State:
            code Bash

        IGNORE_WHEN_COPYING_START
        IGNORE_WHEN_COPYING_END

            
        nmcli device status

          

        Your WIFI_IFACE should show connected and the connection name should be your hotspot's name (e.g., Fedora Hotspot). If it shows disconnected or unavailable, try sudo nmcli connection up "Fedora Hotspot".
        Check dmesg | grep -i wifi for any driver-related errors.

        Check NetworkManager Logs: journalctl -u NetworkManager --since "5 minutes ago" can reveal errors during hotspot activation.

        Try Different Channel/Band (Advanced): Rarely, but sometimes channel conflicts or band issues can prevent visibility. NetworkManager usually handles this well. You can try forcing an 802.11g (2.4GHz) or 802.11a/n/ac (5GHz) mode if your card supports both, or a specific channel, but this is usually not required.

Pain Point: Hotspot Creates but Clients Get No IP Address

    Symptom: Clients can see the hotspot and attempt to connect, but they either fail to connect or connect but show "no internet" or a self-assigned IP address (like 169.254.X.X).

    Root Causes:

        NetworkManager's internal DHCP server is not running or misconfigured.

        ipv4.method shared was not correctly applied to the hotspot connection.

        Firewall blocking DHCP traffic on the hotspot interface.

    Resolution:

        Verify shared method:
        code Bash

        IGNORE_WHEN_COPYING_START
        IGNORE_WHEN_COPYING_END

            
        nmcli connection show "Fedora Hotspot" | grep ipv4.method

          

        It must show ipv4.method: shared. If not, re-run sudo nmcli connection modify "Fedora Hotspot" ipv4.method shared and then sudo nmcli connection up "Fedora Hotspot".

        Firewall for Hotspot Zone: While NetworkManager usually handles this, ensure the zone your hotspot interface is in allows DHCP traffic.

            Find the hotspot interface's zone: sudo firewall-cmd --get-active-zones.

            Check if DHCP is allowed: sudo firewall-cmd --zone=<hotspot_zone> --query-service=dhcp.

            If no, add it: sudo firewall-cmd --zone=<hotspot_zone> --add-service=dhcp --permanent && sudo firewall-cmd --reload.

            However, typically NetworkManager's shared mode sets up a specific zone or rules that cover this, so direct manipulation is rarely needed unless you've made custom firewall changes.

        NetworkManager Logs: Check journalctl -u NetworkManager --since "5 minutes ago" for any errors related to DHCP or IP address assignment for the hotspot connection.

Pain Point: Slow Speeds or Instability

    Symptom: Internet connection through the hotspot is noticeably slower or frequently drops for clients.

    Root Causes:

        Poor Wi-Fi signal strength/interference.

        Overloaded system resources on the Fedora machine.

        Outdated Wi-Fi drivers.

        Limited Wi-Fi hardware capabilities.

    Resolution:

        Check Signal/Interference: Move the Fedora laptop closer to client devices. Use a Wi-Fi analyzer app on a phone to check for congested channels in your area.

        Driver Update: Ensure your Fedora system is fully updated (sudo dnf update). Newer kernel versions often bring updated Wi-Fi drivers.

        System Load: Check system resource usage (top, htop, free -h) on the Fedora machine. Heavy CPU or RAM usage could impact network performance.

        USB Wi-Fi Adapter: If the built-in Wi-Fi is consistently problematic, consider a good quality USB Wi-Fi adapter with better antenna design or newer chipset.

Pain Point: Hotspot Disappears After Reboot

    Symptom: Hotspot works fine until reboot, then needs to be reconfigured or manually started.

    Root Causes:

        autoconnect not set for the connection.

        sysctl.conf not properly saved or applied for IP forwarding.

    Resolution:

        Autoconnect for Hotspot: Ensure autoconnect yes is set for your hotspot connection.
        code Bash

            IGNORE_WHEN_COPYING_START
            IGNORE_WHEN_COPYING_END

                
            sudo nmcli connection modify "${HOTSPOT_CONN_NAME}" autoconnect yes

              

            Persistent IP Forwarding: Re-verify net.ipv4.ip_forward = 1 in /etc/sysctl.conf and that sudo sysctl -p was run.

By following these detailed steps and using the troubleshooting guide, you should be able to successfully set up and maintain a stable Wi-Fi hotspot for network sharing from your Fedora 42 system.
